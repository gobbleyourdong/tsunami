Cache stats

Shift-click anywhere to blast a destruction circle. ALL archetypes that overlap (terrain, volcano, rock, balls, the gear, the robot) get only their intersecting sub-region marked dirty and re-baked from the SDF — the SDF is ground truth, the cache is just memo.
Two clocks. Physics + translation run unlocked at the render rate (60 fps — buttery smooth motion). Rotation, scale, and SDF re-bakes lock to the 12 fps animation tick (every LOOP_TICK_INTERVAL_FRAMES = 5 render frames — chunky retro feel). The yellow border flashes every tick.
Two cache states. static = baked once at init, never touched again. animating = single buffer re-baked from the SDF on tick boundaries.
1-second loop quantization. Each animating archetype declares loopFrames ∈ {1,2,3,4,6,12} (must divide 12). The full loop always plays in exactly 1 second; each frame holds for 12 / loopFrames anim ticks. So a 4-frame walk cycle is 4 fps (3 ticks per frame, very chunky), a 12-frame loop is 12 fps (1 tick per frame, smoothest). The re-bake fires only when the loop's frame index changes — a 4fps loop only bakes 4 times per second, not 12. Same loopTick offset → same buffer, deterministically.
Watch the critter slide smoothly across the screen (translation = per-frame) while its limbs snap into new walk poses on the tick (deformation = per-tick). Watch the fan rotate in 30° steps every tick, not buttery. The heart and gear pulse/wiggle on the same beat.
Per-frame work: pure drawImage from cached buffers + position update. SDF evals = 0 between ticks. On tick: re-bake every animating archetype. The HUD's "bakes last tick" is the exact count.
Toggle depth or normal to see the parallel channels baked alongside color — the renderer uses them for z-sorting and relighting without ever re-running the SDF.
